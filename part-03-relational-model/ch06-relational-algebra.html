6  Relational Algebra

Chapter 6

SQL is a declarative language: you say what you want; the DBMS decides how to compute it. The “how” is determined by translating your SQL into relational algebra — a procedural formal language where every operation is an explicit step. Understanding relational algebra means understanding how the query optimizer thinks, why certain queries are fast, and what is happening beneath every JOIN you write.

6.1.1 Entity-Relationship Overview

6.2 What Is Relational Algebra?

Relational algebra is a procedural formal query language for relations. Each operation takes one or two relations as input and returns a new relation as output.

Closure property: The result of every relational algebra operation is itself a relation. This means operations can be freely composed — the output of one becomes the input of the next.

\text{Expression} = \text{Op}_k(\ldots\text{Op}_2(\text{Op}_1(R_1, R_2))\ldots)

There are 6 fundamental operations (from which all others can be derived) and several convenient additional (derived) operations.

Group Operations
Fundamental \sigma select, \pi project, \cup union, - difference, \times Cartesian product, \rho rename
Derived \cap intersection, \bowtie natural join, \bowtie_\theta theta join, \ltimes outer joins, \div division

6.3 Result Schema Notation

For every operation in this chapter, we track the result schema — the column names and degree of the output relation. This is exactly what the DBMS query optimizer computes before execution.

Format used:

Result: RelationName(col1, col2, …)
Degree: k    Cardinality: m rows (estimated)

Column names in a result use qualified notation Table.Column when there is ambiguity (e.g., after a Cartesian product that produces two Ssn columns).

6.4 Fundamental Operations

6.4.1 1. Select — \sigma (Filter Rows)

\sigma_{\text{condition}}(R)

Returns the subset of tuples in R that satisfy the predicate. Equivalent to a SQL WHERE clause.

Properties: - Degree of result = degree of R (same columns) - Cardinality of result ≤ cardinality of R - Conditions: =, \neq, <, >, \leq, \geq, combined with \land (AND), \lor (OR), \lnot (NOT)

  • LaTeX Notation
  • RELAX Input

\sigma_{\text{Salary} > 65000}(\text{Employee})

σ Salary > 65000 (Employee)

Result Schema:

Result: Employee(Fname, Minit, Lname, Ssn, Bdate, Address, Gender, Salary, Super_ssn, Dno)
Degree: 10
Cardinality: ~12 rows (employees earning > 65,000)

Compound example — employees in department 1 earning more than 60,000:

  • LaTeX Notation
  • RELAX Input

\sigma_{\text{Dno}=1 \;\land\; \text{Salary}>60000}(\text{Employee})

σ Dno = 1 and Salary > 60000 (Employee)

SQL equivalent:

SELECT * FROM Employee WHERE Dno = 1 AND Salary > 60000;

6.4.2 2. Project — \pi (Filter Columns)

\pi_{A_1, A_2, \ldots, A_k}(R)

Returns a relation containing only the listed attributes. Duplicate tuples are eliminated (set semantics).

Properties: - Degree of result = k (number of listed attributes) - Cardinality ≤ cardinality of R (duplicates eliminated)

  • LaTeX Notation
  • RELAX Input

\pi_{\text{Fname, Lname, Salary}}(\text{Employee})

π Fname, Lname, Salary (Employee)

Result Schema:

Result: (Fname, Lname, Salary)
Degree: 3    Cardinality: ≤ 30 rows
Tip

The SQL keyword DISTINCT corresponds to projection in the formal model. Without DISTINCT, SQL is a bag (multiset) operation; with it, SQL matches formal set semantics.

6.4.3 3. Union — \cup (Combine Two Relations)

R \cup S

Returns all tuples in R, all tuples in S, with duplicates removed.

Set Compatibility Rules

\cup, \cap, and - all require union-compatible operands:

  1. R and S must have the same degree (number of attributes)
  2. Corresponding attributes must have compatible domains (same or convertible types)
  3. The result takes the attribute names from the left operand (R)
  • LaTeX Notation
  • RELAX Input

\pi_{\text{Ssn}}(\text{Employee}) \;\cup\; \pi_{\text{Essn}}(\text{Works\_On})

(π Ssn (Employee)) ∪ (π Essn (Works_On))

Result Schema:

Result: (Ssn)          ← name taken from LEFT operand
Degree: 1
Cardinality: ≤ 30 + 20 = 50 (minus duplicates)

SQL equivalent: UNION (removes duplicates); UNION ALL (keeps duplicates → bag semantics).

6.4.4 4. Set Difference — - (In One But Not the Other)

R - S

Returns tuples that are in R but not in S. Requires union compatibility.

  • LaTeX Notation
  • RELAX Input

\pi_{\text{Ssn}}(\text{Employee}) \;-\; \pi_{\text{Essn}}(\text{Works\_On})

(π Ssn (Employee)) - (π Essn (Works_On))

This returns the SSNs of employees who are not currently assigned to any project.

Result Schema:

Result: (Ssn)          ← name from LEFT operand
Degree: 1
Cardinality: ≤ 30 (employees not in Works_On)

SQL equivalent: EXCEPT (standard) or NOT EXISTS / NOT IN subquery.

6.4.5 5. Cartesian Product — \times (All Combinations)

R \times S

Returns every possible combination of one tuple from R with one tuple from S.

Properties: - Degree = degree(R) + degree(S) — columns from both; qualified as R.col and S.col for shared names - Cardinality = |R| \times |S|

  • LaTeX Notation
  • RELAX Input

\text{Employee} \times \text{Department}

Employee × Department

Result Schema:

Result: (Employee.Fname, …, Employee.Dno, Department.Dname, …, Department.Dnumber, …)
Degree: 10 + 4 = 14    Cardinality: 30 × 8 = 240 rows
Important

The raw Cartesian product generates 240 rows but most are meaningless combinations. It is almost always followed by a \sigma to keep only matching rows — that combination is precisely a join.

6.4.6 6. Rename — \rho (Rename Relation or Attributes)

\rho_{S(B_1, B_2, \ldots, B_n)}(R)

Renames relation R to S and its attributes to B_1, \ldots, B_n.

  • LaTeX Notation
  • RELAX Input

\rho_{\text{Supervisor}(\text{Ssn, FName, LName})}(\pi_{\text{Ssn, Fname, Lname}}(\text{Employee}))

ρ Supervisor(Ssn, FName, LName) (π Ssn, Fname, Lname (Employee))

This is especially useful for self-joins (joining a table with itself), where you need two distinct copies with different names.

6.5 Additional (Derived) Operations

6.5.1 7. Intersection — \cap

R \cap S = R - (R - S)

Returns tuples present in both R and S. Requires union compatibility.

  • LaTeX Notation
  • RELAX Input

\pi_{\text{Ssn}}(\text{Employee}) \;\cap\; \pi_{\text{Essn}}(\text{Works\_On})

(π Ssn (Employee)) ∩ (π Essn (Works_On))

Returns SSNs of employees who are assigned to at least one project.

6.5.2 8. Natural Join — \bowtie

R \bowtie S

A natural join automatically: 1. Finds all columns with identical names in both R and S (the common attributes) 2. Keeps only rows where those common attributes have equal values 3. Eliminates the duplicate common columns from the result (they appear once)

Column collapse rule: If R has columns (A, B, C) and S has columns (B, C, D), then R \bowtie S has columns (A, B, C, D) — not (A, B, C, B, C, D).

  • LaTeX Notation
  • RELAX Input

\text{Employee} \bowtie \text{Works\_On}

Employee ⋈ Works_On

Result Schema:

Employee(Fname,…,Ssn,…,Dno) ⋈ Works_On(Essn, Pno, Hours)
Common attribute: Ssn=Essn → joined on equality; one column kept
Result: (Fname, Minit, Lname, Ssn, Bdate, Address, Gender, Salary, Super_ssn, Dno, Pno, Hours)
Degree: 10 + 3 - 1 = 12    Cardinality: ≤ 20 rows (Works_On has 20 rows)
Note

In the Company Database, Employee.Ssn and Works_On.Essn are the intended join columns but have different names. RELAX’s natural join will not match them automatically. Use a theta join with the explicit condition Ssn = Essn, or first rename one column.

6.5.3 9. Theta Join — \bowtie_\theta (Join on Any Condition)

R \bowtie_\theta S = \sigma_\theta(R \times S)

A theta join is a Cartesian product filtered by condition \theta. When \theta uses only equality (=), it is an equijoin.

  • LaTeX Notation
  • RELAX Input

\text{Employee} \bowtie_{\text{Ssn}=\text{Essn}} \text{Works\_On}

Employee ⋈ Ssn = Essn Works_On

Result Schema:

Result: (Employee.Fname,…,Employee.Ssn,…,Works_On.Essn, Works_On.Pno, Works_On.Hours)
Degree: 13    (both Ssn and Essn are kept — unlike natural join)
Cardinality: 20 rows

6.5.4 10. Outer Joins

Standard (inner) joins discard unmatched rows. Outer joins preserve them by filling missing values with NULL.

Operator Symbol Keeps unmatched from
Left outer join R \mathbin{\text{⟕}} S R (left operand)
Right outer join R \mathbin{\text{⟖}} S S (right operand)
Full outer join R \mathbin{\text{⟗}} S Both R and S
  • LaTeX Notation
  • RELAX Input

\text{Employee} \mathbin{\text{⟕}}_{\text{Ssn}=\text{Essn}} \text{Works\_On}

Employee ⟕ Ssn = Essn Works_On

Returns all 30 employees; those not in Works_On get NULL for Pno and Hours.

6.5.5 11. Division — \div (Universal Quantification)

R(A, B) \div S(B)

Returns the values of A in R that are associated with every value of B in S.

R \div S \;\equiv\; \pi_A(R) - \pi_A\bigl((\pi_A(R) \times S) - R\bigr)

Use case: “Find employee SSNs who work on all projects in a given set.”

  • LaTeX Notation
  • RELAX Input

\pi_{\text{Essn, Pno}}(\text{Works\_On}) \;\div\; \pi_{\text{Pnumber}}(\sigma_{\text{Dnum}=1}(\text{Project}))

(π Essn, Pno (Works_On)) ÷ (π Pnumber (σ Dnum = 1 (Project)))

Returns SSNs of employees who work on all projects controlled by department 1.

6.6 Composed Queries — Company Database Examples

6.6.1 Q1: Names of employees who work in the ‘Research’ department

Step 1: Find the department number for ‘Research’: D \leftarrow \sigma_{\text{Dname}='\text{Research}'}(\text{Department})

Step 2: Find employees in that department: E \leftarrow \text{Employee} \bowtie_{\text{Dno}=\text{Dnumber}} D

Step 3: Project names: \pi_{\text{Fname, Lname}}(E)

Composed in one expression: \pi_{\text{Fname, Lname}}\Bigl(\text{Employee} \bowtie_{\text{Dno}=\text{Dnumber}} \sigma_{\text{Dname}='\text{Research}'}(\text{Department})\Bigr)

  • RELAX Input
  • SQL Equivalent
π Fname, Lname (Employee ⋈ Dno = Dnumber (σ Dname = 'Research' (Department)))
SELECT Fname, Lname
FROM Employee
JOIN Department ON Dno = Dnumber
WHERE Dname = 'Research';

6.6.2 Q2: Names and hours of employees working more than 30 hours on any project

\pi_{\text{Fname, Lname, Hours}}\Bigl(\text{Employee} \bowtie_{\text{Ssn}=\text{Essn}} \sigma_{\text{Hours}>30}(\text{Works\_On})\Bigr)

  • RELAX Input
  • SQL Equivalent
π Fname, Lname, Hours (Employee ⋈ Ssn = Essn (σ Hours > 30 (Works_On)))
SELECT Fname, Lname, Hours
FROM Employee
JOIN Works_On ON Ssn = Essn
WHERE Hours > 30;

6.6.3 Q3: Employees who do NOT work on any project

\pi_{\text{Ssn}}(\text{Employee}) - \pi_{\text{Essn}}(\text{Works\_On})

Then join back to get names: \pi_{\text{Fname, Lname}}\Bigl(\text{Employee} \bowtie_{\text{Ssn}=\text{Ssn}} \bigl(\pi_{\text{Ssn}}(\text{Employee}) - \pi_{\text{Essn}}(\text{Works\_On})\bigr)\Bigr)

  • RELAX Input
  • SQL Equivalent
π Fname, Lname (Employee ⋈ Ssn = Ssn ((π Ssn (Employee)) - (π Essn (Works_On))))
SELECT Fname, Lname
FROM Employee
WHERE Ssn NOT IN (SELECT Essn FROM Works_On);

6.7 Algebra → Execution Plan and Query Optimization

When the DBMS receives a SQL query, the DML compiler translates it into a relational algebra expression tree. The query optimizer then applies equivalence rules to rewrite the tree into a cheaper form.

6.7.1 Expression Trees

An expression tree (also called a query tree) is a binary tree where: - Leaf nodes are operand relations (base tables) - Internal nodes are operators (\sigma, \pi, \bowtie, …) - Data flows bottom-up: leaves feed into parent operators

Example tree for Q1:

Notice that \sigma is applied before the join — filtering Department from 8 rows down to 1 row before the expensive join operation.

6.7.2 Equivalence Rules

Rule Statement Optimization impact
R1 Cascade select \sigma_{c_1 \land c_2}(R) \equiv \sigma_{c_1}(\sigma_{c_2}(R)) Apply each condition separately
R2 Commutativity of select \sigma_{c_1}(\sigma_{c_2}(R)) \equiv \sigma_{c_2}(\sigma_{c_1}(R)) Apply cheapest condition first
R3 Cascade project \pi_{L}(\pi_{L'}(R)) \equiv \pi_{L}(R) if L \subseteq L' Eliminate intermediate projections
R4 Push select past join \sigma_c(R \bowtie S) \equiv \sigma_c(R) \bowtie S if c involves only R Reduce join inputs
R5 Commutativity of join R \bowtie S \equiv S \bowtie R Put smaller relation on the right (build side)
R6 Associativity of join (R \bowtie S) \bowtie T \equiv R \bowtie (S \bowtie T) Reorder joins for minimal intermediate size

6.7.3 Cost Estimation

The optimizer assigns a cost estimate to each plan. The dominant cost factor is the number of disk I/O operations (page reads). A simple model:

\text{cost}(\sigma_c(R)) = b_R \;\text{ page reads}

\text{cost}(R \bowtie_\theta S) \approx b_R + |R| \cdot b_S \;\text{ (nested-loop join, worst case)}

where b_R = number of disk pages in R and |R| = cardinality of R.

Pushing a \sigma before a join reduces |R| (and hence b_R), making the join substantially cheaper.

6.7.4 Before-and-After Optimization Example

Naive plan (SQL compiler’s initial translation):

π Fname, Lname (σ Dname='Research' (Employee × Department))

Cost: 30 \times 8 = 240 Cartesian product rows, then filter.

Optimized plan (after applying R4 and R1):

π Fname, Lname (Employee ⋈_{Dno=Dnumber} (σ Dname='Research' (Department)))

Cost: Filter 8 department rows → 1 row → join 30 employees against 1 row = 30 comparisons.

The optimization reduces tuples processed from 240 to 30 — and that ratio only grows with larger tables.

6.8 Relational Algebra vs. SQL — Correspondence

Relational Algebra SQL Equivalent
\sigma_c(R) SELECT * FROM R WHERE c
\pi_{a,b}(R) SELECT DISTINCT a, b FROM R
R \cup S … UNION …
R - S … EXCEPT … (or NOT EXISTS)
R \times S FROM R CROSS JOIN S
R \bowtie S FROM R NATURAL JOIN S
R \bowtie_\theta S FROM R JOIN S ON θ
R \mathbin{\text{⟕}}_\theta S FROM R LEFT OUTER JOIN S ON θ
R \div S No direct SQL; simulated with NOT EXISTS double negation
\rho_S(R) … AS alias …

6.9 Summary

ImportantChapter 6 — Key Takeaways
  • Relational algebra is procedural, closed, and composable. Every SQL query translates to a RA expression internally.
  • 6 fundamental operations: \sigma (filter rows), \pi (filter columns), \cup (union), - (difference), \times (Cartesian product), \rho (rename).
  • Set compatibility for \cup, \cap, -: same degree + compatible domains; result takes left operand’s column names.
  • Natural join auto-matches on identically-named columns and collapses duplicates to one column. Always verify shared column names match your intent.
  • Theta join = \sigma_\theta(R \times S); equijoin uses only =; natural join is a special equijoin that removes duplicate columns.
  • Outer joins preserve unmatched tuples by padding with NULLs.
  • Division (÷) answers universal queries: “find tuples associated with all tuples in S”.
  • Result schema tracking: every operation has a known degree, known column names (qualified if needed), and an estimated cardinality.
  • Query optimization: expression trees + 6 equivalence rules. Key rule: always push \sigma as far down the tree as possible to minimize join input sizes.

## What Is Relational Algebra?

**Relational algebra** is a **procedural formal query language** for relations. Each *operation* takes one or two relations as input and returns a new relation as output.

**Closure property**: The result of every relational algebra operation is itself a relation. This means operations can be freely composed — the output of one becomes the input of the next.

$$\text{Expression} = \text{Op}_k(\ldots\text{Op}_2(\text{Op}_1(R_1, R_2))\ldots)$$

There are **6 fundamental operations** (from which all others can be derived) and several convenient **additional (derived) operations**.

| Group | Operations |
|-------|-----------|
| Fundamental | $\sigma$ select, $\pi$ project, $\cup$ union, $-$ difference, $\times$ Cartesian product, $\rho$ rename |
| Derived | $\cap$ intersection, $\bowtie$ natural join, $\bowtie_\theta$ theta join, $\ltimes$ outer joins, $\div$ division |

---

## Result Schema Notation

For every operation in this chapter, we track the **result schema** — the column names and degree of the output relation. This is exactly what the DBMS query optimizer computes before execution.

**Format used:**
```
Result: RelationName(col1, col2, …)
Degree: k    Cardinality: m rows (estimated)
```

Column names in a result use **qualified notation** `Table.Column` when there is ambiguity (e.g., after a Cartesian product that produces two `Ssn` columns).

---

## Fundamental Operations

### 1. Select — $\sigma$ (Filter Rows)

$$\sigma_{\text{condition}}(R)$$

Returns the subset of tuples in $R$ that satisfy the *predicate*. Equivalent to a SQL `WHERE` clause.

**Properties:**
- Degree of result = degree of $R$ (same columns)
- Cardinality of result ≤ cardinality of $R$
- Conditions: $=, \neq, <, >, \leq, \geq$, combined with $\land$ (AND), $\lor$ (OR), $\lnot$ (NOT)

::: {.panel-tabset}
## LaTeX Notation
$$\sigma_{\text{Salary} > 65000}(\text{Employee})$$

## RELAX Input
```
σ Salary > 65000 (Employee)
```
:::

**Result Schema:**
```
Result: Employee(Fname, Minit, Lname, Ssn, Bdate, Address, Gender, Salary, Super_ssn, Dno)
Degree: 10
Cardinality: ~12 rows (employees earning > 65,000)
```

**Compound example** — employees in department 1 earning more than 60,000:

::: {.panel-tabset}
## LaTeX Notation
$$\sigma_{\text{Dno}=1 \;\land\; \text{Salary}>60000}(\text{Employee})$$

## RELAX Input
```
σ Dno = 1 and Salary > 60000 (Employee)
```
:::

**SQL equivalent:**
```sql
SELECT * FROM Employee WHERE Dno = 1 AND Salary > 60000;
```

---

### 2. Project — $\pi$ (Filter Columns)

$$\pi_{A_1, A_2, \ldots, A_k}(R)$$

Returns a relation containing only the listed attributes. Duplicate tuples are **eliminated** (set semantics).

**Properties:**
- Degree of result = $k$ (number of listed attributes)
- Cardinality ≤ cardinality of $R$ (duplicates eliminated)

::: {.panel-tabset}
## LaTeX Notation
$$\pi_{\text{Fname, Lname, Salary}}(\text{Employee})$$

## RELAX Input
```
π Fname, Lname, Salary (Employee)
```
:::

**Result Schema:**
```
Result: (Fname, Lname, Salary)
Degree: 3    Cardinality: ≤ 30 rows
```

::: callout-tip
The SQL keyword `DISTINCT` corresponds to projection in the formal model. Without `DISTINCT`, SQL is a **bag** (multiset) operation; with it, SQL matches formal **set** semantics.
:::

---

### 3. Union — $\cup$ (Combine Two Relations)

$$R \cup S$$

Returns all tuples in $R$, all tuples in $S$, with **duplicates removed**.

#### Set Compatibility Rules

$\cup$, $\cap$, and $-$ all require **union-compatible** operands:

1. $R$ and $S$ must have the **same degree** (number of attributes)
2. Corresponding attributes must have **compatible domains** (same or convertible types)
3. The **result takes the attribute names from the left operand** ($R$)

::: {.panel-tabset}
## LaTeX Notation
$$\pi_{\text{Ssn}}(\text{Employee}) \;\cup\; \pi_{\text{Essn}}(\text{Works\_On})$$

## RELAX Input
```
(π Ssn (Employee)) ∪ (π Essn (Works_On))
```
:::

**Result Schema:**
```
Result: (Ssn)          ← name taken from LEFT operand
Degree: 1
Cardinality: ≤ 30 + 20 = 50 (minus duplicates)
```

**SQL equivalent:** `UNION` (removes duplicates); `UNION ALL` (keeps duplicates → bag semantics).

---

### 4. Set Difference — $-$ (In One But Not the Other)

$$R - S$$

Returns tuples that are in $R$ but **not** in $S$. Requires union compatibility.

::: {.panel-tabset}
## LaTeX Notation
$$\pi_{\text{Ssn}}(\text{Employee}) \;-\; \pi_{\text{Essn}}(\text{Works\_On})$$

## RELAX Input
```
(π Ssn (Employee)) - (π Essn (Works_On))
```
:::

This returns the SSNs of employees who are **not** currently assigned to any project.

**Result Schema:**
```
Result: (Ssn)          ← name from LEFT operand
Degree: 1
Cardinality: ≤ 30 (employees not in Works_On)
```

**SQL equivalent:** `EXCEPT` (standard) or `NOT EXISTS` / `NOT IN` subquery.

---

### 5. Cartesian Product — $\times$ (All Combinations)

$$R \times S$$

Returns every possible combination of one tuple from $R$ with one tuple from $S$.

**Properties:**
- Degree = degree($R$) + degree($S$) — columns from both; qualified as `R.col` and `S.col` for shared names
- Cardinality = $|R| \times |S|$

::: {.panel-tabset}
## LaTeX Notation
$$\text{Employee} \times \text{Department}$$

## RELAX Input
```
Employee × Department
```
:::

**Result Schema:**
```
Result: (Employee.Fname, …, Employee.Dno, Department.Dname, …, Department.Dnumber, …)
Degree: 10 + 4 = 14    Cardinality: 30 × 8 = 240 rows
```

::: callout-important
The raw Cartesian product generates 240 rows but most are meaningless combinations. It is almost always followed by a $\sigma$ to keep only matching rows — that combination is precisely a **join**.
:::

---

### 6. Rename — $\rho$ (Rename Relation or Attributes)

$$\rho_{S(B_1, B_2, \ldots, B_n)}(R)$$

Renames relation $R$ to $S$ and its attributes to $B_1, \ldots, B_n$.

::: {.panel-tabset}
## LaTeX Notation
$$\rho_{\text{Supervisor}(\text{Ssn, FName, LName})}(\pi_{\text{Ssn, Fname, Lname}}(\text{Employee}))$$

## RELAX Input
```
ρ Supervisor(Ssn, FName, LName) (π Ssn, Fname, Lname (Employee))
```
:::

This is especially useful for **self-joins** (joining a table with itself), where you need two distinct copies with different names.

---

## Additional (Derived) Operations

### 7. Intersection — $\cap$

$$R \cap S = R - (R - S)$$

Returns tuples present in **both** $R$ and $S$. Requires union compatibility.

::: {.panel-tabset}
## LaTeX Notation
$$\pi_{\text{Ssn}}(\text{Employee}) \;\cap\; \pi_{\text{Essn}}(\text{Works\_On})$$

## RELAX Input
```
(π Ssn (Employee)) ∩ (π Essn (Works_On))
```
:::

Returns SSNs of employees who are assigned to at least one project.

---

### 8. Natural Join — $\bowtie$

$$R \bowtie S$$

A natural join automatically:
1. Finds all columns with **identical names** in both $R$ and $S$ (the common attributes)
2. Keeps only rows where those common attributes have **equal values**
3. **Eliminates the duplicate common columns** from the result (they appear once)

**Column collapse rule:** If $R$ has columns $(A, B, C)$ and $S$ has columns $(B, C, D)$, then $R \bowtie S$ has columns $(A, B, C, D)$ — not $(A, B, C, B, C, D)$.

::: {.panel-tabset}
## LaTeX Notation
$$\text{Employee} \bowtie \text{Works\_On}$$

## RELAX Input
```
Employee ⋈ Works_On
```
:::

**Result Schema:**
```
Employee(Fname,…,Ssn,…,Dno) ⋈ Works_On(Essn, Pno, Hours)
Common attribute: Ssn=Essn → joined on equality; one column kept
Result: (Fname, Minit, Lname, Ssn, Bdate, Address, Gender, Salary, Super_ssn, Dno, Pno, Hours)
Degree: 10 + 3 - 1 = 12    Cardinality: ≤ 20 rows (Works_On has 20 rows)
```

::: callout-note
In the Company Database, `Employee.Ssn` and `Works_On.Essn` are the **intended** join columns but have *different names*. RELAX's natural join will not match them automatically. Use a **theta join** with the explicit condition `Ssn = Essn`, or first rename one column.
:::

---

### 9. Theta Join — $\bowtie_\theta$ (Join on Any Condition)

$$R \bowtie_\theta S = \sigma_\theta(R \times S)$$

A theta join is a Cartesian product filtered by condition $\theta$. When $\theta$ uses only equality (`=`), it is an **equijoin**.

::: {.panel-tabset}
## LaTeX Notation
$$\text{Employee} \bowtie_{\text{Ssn}=\text{Essn}} \text{Works\_On}$$

## RELAX Input
```
Employee ⋈ Ssn = Essn Works_On
```
:::

**Result Schema:**
```
Result: (Employee.Fname,…,Employee.Ssn,…,Works_On.Essn, Works_On.Pno, Works_On.Hours)
Degree: 13    (both Ssn and Essn are kept — unlike natural join)
Cardinality: 20 rows
```

---

### 10. Outer Joins

Standard (inner) joins discard unmatched rows. **Outer joins** preserve them by filling missing values with NULL.

| Operator | Symbol | Keeps unmatched from |
|----------|--------|---------------------|
| Left outer join | $R \mathbin{\text{⟕}} S$ | $R$ (left operand) |
| Right outer join | $R \mathbin{\text{⟖}} S$ | $S$ (right operand) |
| Full outer join | $R \mathbin{\text{⟗}} S$ | Both $R$ and $S$ |

::: {.panel-tabset}
## LaTeX Notation
$$\text{Employee} \mathbin{\text{⟕}}_{\text{Ssn}=\text{Essn}} \text{Works\_On}$$

## RELAX Input
```
Employee ⟕ Ssn = Essn Works_On
```
:::

Returns all 30 employees; those not in `Works_On` get NULL for `Pno` and `Hours`.

---

### 11. Division — $\div$ (Universal Quantification)

$$R(A, B) \div S(B)$$

Returns the values of $A$ in $R$ that are associated with **every** value of $B$ in $S$.

$$R \div S \;\equiv\; \pi_A(R) - \pi_A\bigl((\pi_A(R) \times S) - R\bigr)$$

**Use case:** "Find employee SSNs who work on **all** projects in a given set."

::: {.panel-tabset}
## LaTeX Notation
$$\pi_{\text{Essn, Pno}}(\text{Works\_On}) \;\div\; \pi_{\text{Pnumber}}(\sigma_{\text{Dnum}=1}(\text{Project}))$$

## RELAX Input
```
(π Essn, Pno (Works_On)) ÷ (π Pnumber (σ Dnum = 1 (Project)))
```
:::

Returns SSNs of employees who work on **all** projects controlled by department 1.

## Composed Queries — Company Database Examples

### Q1: Names of employees who work in the 'Research' department

**Step 1:** Find the department number for 'Research':
$$D \leftarrow \sigma_{\text{Dname}='\text{Research}'}(\text{Department})$$

**Step 2:** Find employees in that department:
$$E \leftarrow \text{Employee} \bowtie_{\text{Dno}=\text{Dnumber}} D$$

**Step 3:** Project names:
$$\pi_{\text{Fname, Lname}}(E)$$

**Composed in one expression:**
$$\pi_{\text{Fname, Lname}}\Bigl(\text{Employee} \bowtie_{\text{Dno}=\text{Dnumber}} \sigma_{\text{Dname}='\text{Research}'}(\text{Department})\Bigr)$$

::: {.panel-tabset}
## RELAX Input
```
π Fname, Lname (Employee ⋈ Dno = Dnumber (σ Dname = 'Research' (Department)))
```

## SQL Equivalent
```sql
SELECT Fname, Lname
FROM Employee
JOIN Department ON Dno = Dnumber
WHERE Dname = 'Research';
```
:::

---

### Q2: Names and hours of employees working more than 30 hours on any project

$$\pi_{\text{Fname, Lname, Hours}}\Bigl(\text{Employee} \bowtie_{\text{Ssn}=\text{Essn}} \sigma_{\text{Hours}>30}(\text{Works\_On})\Bigr)$$

::: {.panel-tabset}
## RELAX Input
```
π Fname, Lname, Hours (Employee ⋈ Ssn = Essn (σ Hours > 30 (Works_On)))
```

## SQL Equivalent
```sql
SELECT Fname, Lname, Hours
FROM Employee
JOIN Works_On ON Ssn = Essn
WHERE Hours > 30;
```
:::

---

### Q3: Employees who do NOT work on any project

$$\pi_{\text{Ssn}}(\text{Employee}) - \pi_{\text{Essn}}(\text{Works\_On})$$

Then join back to get names:
$$\pi_{\text{Fname, Lname}}\Bigl(\text{Employee} \bowtie_{\text{Ssn}=\text{Ssn}} \bigl(\pi_{\text{Ssn}}(\text{Employee}) - \pi_{\text{Essn}}(\text{Works\_On})\bigr)\Bigr)$$

::: {.panel-tabset}
## RELAX Input
```
π Fname, Lname (Employee ⋈ Ssn = Ssn ((π Ssn (Employee)) - (π Essn (Works_On))))
```

## SQL Equivalent
```sql
SELECT Fname, Lname
FROM Employee
WHERE Ssn NOT IN (SELECT Essn FROM Works_On);
```
:::

## Algebra → Execution Plan and Query Optimization

When the DBMS receives a SQL query, the **DML compiler** translates it into a relational algebra expression tree. The **query optimizer** then applies **equivalence rules** to rewrite the tree into a cheaper form.

### Expression Trees

An **expression tree** (also called a *query tree*) is a binary tree where:
- **Leaf nodes** are operand relations (base tables)
- **Internal nodes** are operators ($\sigma$, $\pi$, $\bowtie$, …)
- Data flows **bottom-up**: leaves feed into parent operators

**Example tree for Q1:**

```{mermaid}
%%| eval: true
%%| echo: false
flowchart TD
    PI["π Fname, Lname"]
    J["⋈ Dno = Dnumber"]
    EMP["Employee<br/>(30 rows)"]
    S["σ Dname = 'Research'"]
    DEPT["Department<br/>(8 rows)"]

    PI --> J
    J --> EMP
    J --> S
    S --> DEPT
```

Notice that $\sigma$ is applied **before** the join — filtering `Department` from 8 rows down to 1 row before the expensive join operation.

### Equivalence Rules

| Rule | Statement | Optimization impact |
|------|-----------|-------------------|
| **R1** Cascade select | $\sigma_{c_1 \land c_2}(R) \equiv \sigma_{c_1}(\sigma_{c_2}(R))$ | Apply each condition separately |
| **R2** Commutativity of select | $\sigma_{c_1}(\sigma_{c_2}(R)) \equiv \sigma_{c_2}(\sigma_{c_1}(R))$ | Apply cheapest condition first |
| **R3** Cascade project | $\pi_{L}(\pi_{L'}(R)) \equiv \pi_{L}(R)$ if $L \subseteq L'$ | Eliminate intermediate projections |
| **R4** Push select past join | $\sigma_c(R \bowtie S) \equiv \sigma_c(R) \bowtie S$ if $c$ involves only $R$ | Reduce join inputs |
| **R5** Commutativity of join | $R \bowtie S \equiv S \bowtie R$ | Put smaller relation on the right (build side) |
| **R6** Associativity of join | $(R \bowtie S) \bowtie T \equiv R \bowtie (S \bowtie T)$ | Reorder joins for minimal intermediate size |

### Cost Estimation

The optimizer assigns a **cost estimate** to each plan. The dominant cost factor is the number of **disk I/O operations** (page reads). A simple model:

$$\text{cost}(\sigma_c(R)) = b_R \;\text{ page reads}$$

$$\text{cost}(R \bowtie_\theta S) \approx b_R + |R| \cdot b_S \;\text{ (nested-loop join, worst case)}$$

where $b_R$ = number of disk pages in $R$ and $|R|$ = cardinality of $R$.

Pushing a $\sigma$ before a join reduces $|R|$ (and hence $b_R$), making the join substantially cheaper.

### Before-and-After Optimization Example

**Naive plan** (SQL compiler's initial translation):
```
π Fname, Lname (σ Dname='Research' (Employee × Department))
```
Cost: $30 \times 8 = 240$ Cartesian product rows, then filter.

**Optimized plan** (after applying R4 and R1):
```
π Fname, Lname (Employee ⋈_{Dno=Dnumber} (σ Dname='Research' (Department)))
```
Cost: Filter 8 department rows → 1 row → join 30 employees against 1 row = 30 comparisons.

The optimization reduces tuples processed from **240 to 30** — and that ratio only grows with larger tables.

## Relational Algebra vs. SQL — Correspondence

| Relational Algebra | SQL Equivalent |
|-------------------|----------------|
| $\sigma_c(R)$ | `SELECT * FROM R WHERE c` |
| $\pi_{a,b}(R)$ | `SELECT DISTINCT a, b FROM R` |
| $R \cup S$ | `… UNION …` |
| $R - S$ | `… EXCEPT …` (or `NOT EXISTS`) |
| $R \times S$ | `FROM R CROSS JOIN S` |
| $R \bowtie S$ | `FROM R NATURAL JOIN S` |
| $R \bowtie_\theta S$ | `FROM R JOIN S ON θ` |
| $R \mathbin{\text{⟕}}_\theta S$ | `FROM R LEFT OUTER JOIN S ON θ` |
| $R \div S$ | No direct SQL; simulated with `NOT EXISTS` double negation |
| $\rho_S(R)$ | `… AS alias …` |

## Summary

::: callout-important
## Chapter 6 — Key Takeaways

- **Relational algebra** is procedural, closed, and composable. Every SQL query translates to a RA expression internally.
- **6 fundamental operations**: $\sigma$ (filter rows), $\pi$ (filter columns), $\cup$ (union), $-$ (difference), $\times$ (Cartesian product), $\rho$ (rename).
- **Set compatibility** for $\cup$, $\cap$, $-$: same degree + compatible domains; result takes *left* operand's column names.
- **Natural join** auto-matches on identically-named columns and **collapses duplicates** to one column. Always verify shared column names match your intent.
- **Theta join** = $\sigma_\theta(R \times S)$; equijoin uses only `=`; natural join is a special equijoin that removes duplicate columns.
- **Outer joins** preserve unmatched tuples by padding with NULLs.
- **Division (÷)** answers universal queries: "find tuples associated with *all* tuples in $S$".
- **Result schema tracking**: every operation has a known degree, known column names (qualified if needed), and an estimated cardinality.
- **Query optimization**: expression trees + 6 equivalence rules. Key rule: always push $\sigma$ as far down the tree as possible to minimize join input sizes.
:::
